# Event Deduplication One-Pager (Before vs After)

## Audience and goal

This note explains, in plain English, how event deduplication worked before, what changed, and how event extraction now flows through deduplication.

## Why this change was needed

Before this redesign, deduplication depended too heavily on a fingerprint string generated by the AI model. Even with good prompting, model output can drift over time (format differences, wording differences, missing parts). That made event identity less consistent than we wanted.

In practical terms, similar updates could sometimes be split into separate events just because the fingerprint text varied.

## How dedup worked before

1. The AI returned structured extraction fields plus an `event_fingerprint` string.
2. Validation mainly checked that the fingerprint existed as a string.
3. Backend canonicalization normalized some fields (for example country aliases), but only lightly adjusted the fingerprint if it already matched a very specific format.
4. Event matching primarily used exact fingerprint equality inside a time window.
5. If fingerprint formatting changed (or was malformed), strong matching could fail and duplicates were more likely.

Summary of the old model: the AI was effectively the source of truth for hard event identity.

## What changed

### 1) Backend is now the source of truth for hard identity

The backend now builds the authoritative fingerprint after canonicalization from stable structured fields, in a fixed order, with fixed placeholders, then hashes it.

- Fingerprint versioning was added (`v1`) from day one.
- The AI-provided fingerprint is no longer authoritative for matching.

### 2) Better audit visibility

We now preserve both:

- what the AI proposed (candidate fingerprint), and
- what the backend computed (authoritative fingerprint + version + canonical input string).

This makes debugging and model-quality review much easier.

### 3) Hard vs soft identity is explicitly separated

- Hard identity: deterministic backend fingerprint (for strict dedup).
- Soft identity: contextual similarity (entities, keywords, source, time window).

Fuzzy logic is not mixed into the hard fingerprint itself.

### 4) Safe fallback when identity is too thin

If structured fields are not strong enough for a reliable hard identity, the system does **not** force a possibly wrong fingerprint.

Instead, it skips strict fingerprint matching and relies on contextual matching for that case.

## Current extraction-to-dedup flow

1. A raw message is selected for phase2 extraction.
2. The AI returns structured fields (topic, entities, summary, timing, etc.).
3. Validation accepts missing/null/malformed AI fingerprint candidates without failing identity logic.
4. Canonicalization normalizes fields (countries, entities, tickers, keywords, source text, summary safety rules).
5. Backend computes the authoritative fingerprint (`v1`) from canonicalized fields when identity is sufficient.
6. Extraction is saved with:
   - raw payload,
   - canonical payload,
   - AI fingerprint candidate (if any),
   - backend fingerprint metadata (value/version/canonical input).
7. Dedup decision:
   - If authoritative hard fingerprint exists: try strict match first (fingerprint + time window).
   - If not found, or if no hard fingerprint exists: use contextual soft matching.
8. Event is created or updated, and message-to-event link is stored.

## What this means for product outcomes

- More stable dedup behavior across prompt/model changes.
- Fewer false splits caused by fingerprint formatting drift.
- Clear traceability for "why these messages were clustered".
- Better control of quality over time because fingerprint logic is now owned by backend code, not model wording.

## Known boundary (intentional)

When structured identity is too sparse, the system intentionally avoids forcing a hard fingerprint. This reduces false certainty and keeps matching conservative by falling back to contextual logic.

That tradeoff is expected and safer for production event quality.
